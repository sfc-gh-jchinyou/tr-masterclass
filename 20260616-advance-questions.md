# Advance Questions

| # | Question | Answer |
|---|----------|--------|
| 1 | Would like to have a detailed use case on how to use it and how it could be helpful | A common end-to-end use case: a business analyst asks a natural language question ("What were our top 10 clients by revenue last quarter?") via **Cortex Analyst**. Cortex Analyst uses a **Semantic View** (which defines your business metrics, dimensions, and joins) to generate accurate SQL, runs it against Snowflake tables, and returns results — no SQL knowledge required. This can be embedded in a **Streamlit in Snowflake** app or surfaced through an **Agent** that also pulls in unstructured context from **Cortex Search** (e.g., contract PDFs). The whole pipeline — data, compute, AI, and UI — lives inside Snowflake with no external services needed. |
| 2 | Snowflake MCP server | Snowflake provides a **Model Context Protocol (MCP) server** that lets AI coding assistants (Cursor, VS Code Copilot, Claude Desktop, etc.) interact directly with your Snowflake account. It exposes tools for running SQL, listing objects, reading schemas, and calling Cortex functions — all authenticated through your existing Snowflake credentials. This means your AI assistant can query live data, inspect table structures, and generate grounded SQL without you copy-pasting schemas. Set up via `snow mcp` in the Snowflake CLI. |
| 3 | I'd like to hear about tools you have for troubleshooting data issues | Key tools: **Data Quality Monitoring (DMFs)** — attach metric functions (NULL rate, freshness, uniqueness, custom SQL checks) to tables and schedule them as monitors with alerts. **Snowsight Query History** — inspect execution plans, spill-to-disk warnings, and partition pruning efficiency. **Dynamic Tables** — built-in lag monitoring and `UPSTREAM_FAILED` status for pipeline diagnostics. **Access History & Lineage** in `SNOWFLAKE.ACCOUNT_USAGE` — trace which queries read which columns and build column-level lineage graphs. **Cortex AI functions** (e.g., `AI_FILTER`, `AI_CLASSIFY`) can also flag anomalous or malformed text values at scale. |
| 4 | How do the Snowflake/Cortex tools validate the results? If using multiple sources of data, how does it know which fields to join? | **Cortex Analyst** does not guess joins — it relies entirely on a **Semantic View** (or semantic model YAML) that you define. The semantic view explicitly declares tables, join paths (primary/foreign key relationships), dimensions, measures, and verified query examples. When a natural language question arrives, Cortex Analyst maps it to the defined entities and generates SQL using only the declared joins — it cannot invent a join that isn't in the model. **Verified Query Representations (VQRs)** act as ground-truth examples that further constrain and validate generated SQL. You can evaluate accuracy using the `evaluate_semantic_view` workflow. |
| 5 | How to run notebooks to integrate Python, or to integrate APIs such as Claude | **Snowflake Notebooks** (in Snowsight / Workspaces) support Python cells natively on Snowpark-optimized compute. To call external APIs like Claude (Anthropic): (1) Create an **External Access Integration** + **Network Rule** that allowlists `api.anthropic.com`. (2) Store your API key as a **Snowflake Secret**. (3) In a notebook Python cell, use the standard `requests` or `anthropic` Python library — Snowflake injects the secret and routes traffic through the allowed integration. The same pattern works for any REST API. You can also use `snowflake.cortex.Complete()` directly to call hosted models without any external network access. |
| 6 | Learning more about Cortex would be great | **Snowflake Cortex** is the umbrella for Snowflake's AI/ML services. Key pillars: **Cortex LLM Functions** — `AI_COMPLETE`, `AI_SUMMARIZE_AGG`, `AI_CLASSIFY`, `AI_EXTRACT`, `AI_SENTIMENT`, `AI_TRANSLATE` — serverless, no infra to manage. **Cortex Analyst** — natural language to SQL via semantic views. **Cortex Search** — managed hybrid vector + keyword search over unstructured data (PDFs, emails, docs). **Cortex Agents** — orchestrate multi-step reasoning using tools (Analyst, Search, SQL, external APIs). **Fine-tuning** — adapt hosted models to your domain data. All run inside your Snowflake account; data never leaves your security perimeter. |
| 7 | Would like to know more on agents and semantic layer | **Cortex Agents** are LLM-powered orchestrators that can call multiple tools in sequence: Cortex Analyst (structured data), Cortex Search (unstructured data), SQL execution, and custom tools. You define available tools; the agent decides which to call and in what order based on the user's question. The **Semantic Layer** (Semantic Views) sits beneath Cortex Analyst — it is a YAML/SQL definition of your business model: tables, joins, measures (e.g., `SUM(revenue)`), dimensions, filters, and verified queries. Together, agents + semantic layer enable a "chat with your data" experience that is accurate, governed, and grounded in your actual data model. |
| 8 | Natural language questions of the data and building ad hoc dashboards | **Cortex Analyst** handles natural language → SQL for structured data questions. Results can feed directly into **Snowsight Dashboards** (no-code drag-and-drop charts) or **Streamlit in Snowflake** apps where you write Python to render Altair/Plotly/Matplotlib charts. For fully ad hoc exploration, the Cortex Analyst API returns both the generated SQL and result data, so a Streamlit app can let users type a question, see the SQL, see the chart, and download the data — all in one interface built and hosted inside Snowflake. |
| 9 | Implementation of AI in Snowflake and Cortex integration, and scope of AI tools with Snowflake | Snowflake's AI scope covers: **structured analytics** (Cortex Analyst + Semantic Views), **unstructured data** (Cortex Search, `AI_PARSE_DOCUMENT` for PDFs/images), **text enrichment** (classify, extract, summarize, translate, sentiment on any column), **ML model training & serving** (Snowpark ML, Model Registry, SPCS for custom containers), **agentic workflows** (Cortex Agents), and **fine-tuning** hosted models. The key architectural principle: all AI runs inside your Snowflake account — same governance, RBAC, network policies, and audit logging as your data. No data leaves to a third-party AI service unless you explicitly configure external access. |
| 10 | Will aspects of a self-contained app-building process be covered in the workshop? | Yes — **Streamlit in Snowflake (SiS)** allows you to build and deploy interactive Python web apps entirely within Snowflake (no external hosting, no separate auth). Apps connect to Snowflake via `get_active_session()` and can call Cortex functions, run SQL, and render charts. For more complex full-stack apps, **Snowflake App Runtime** supports Next.js/React apps deployed to SPCS. The workshop will walk through building a Cortex Analyst-powered chat app as a Streamlit app — covering UI, backend query logic, and deployment in one self-contained workflow. |
| 11 | Please start with basics — I am new to TR and not had an opportunity to work on Snowflake | Snowflake is a cloud data platform: you load data into **tables** inside **schemas** inside **databases**. Compute (queries, transformations) runs on **virtual warehouses** (scalable clusters you size and suspend). You interact via **Snowsight** (web UI), SQL, Python (Snowpark), or CLI. Thomson Reuters uses Snowflake to share and analyze data products. In this masterclass, you'll start from the Snowsight UI, write basic SQL, then progressively move into Cortex AI features — no prior Snowflake experience is assumed for the foundational sections. |
| 12 | I want to access and understand pros and cons vs. Claude | **Snowflake Cortex** uses Claude (and other models like Llama, Mistral, Arctic) *under the hood* via `AI_COMPLETE` — so it's not an either/or. The distinction is deployment model: calling **Claude directly** (via Anthropic API) gives you the latest model versions and full flexibility, but your data leaves your environment and you manage auth, rate limits, and costs separately. Using **Claude through Cortex** keeps data inside Snowflake's security perimeter, bills through your Snowflake contract, and integrates with RBAC and audit logging — but you're on the version Snowflake has deployed, not necessarily the latest. For enterprise governed use cases, Cortex is preferred; for bleeding-edge model capabilities, direct API access may be needed. |
| 13 | Understanding what runs the logic behind multi-cluster | A **Multi-Cluster Warehouse** in Snowflake automatically adds or removes warehouse clusters (each cluster is an independent set of compute nodes) based on queue depth and concurrency. The orchestration logic is managed by Snowflake's **Global Services Layer** (the brain of Snowflake, separate from compute and storage). It monitors query queuing; when the current cluster is saturated, it spins up an additional cluster in seconds and routes new queries there. You configure min/max cluster count and scaling policy (Standard = scale out aggressively, Economy = scale out conservatively). There is no user-managed scheduler — Snowflake handles it fully. |
| 14 | Agentic AI and Cortex Integration, Native dbt projects | **Cortex Agents** (see Q7) can be built via the REST API or Python SDK and deployed as Streamlit apps or called from external systems. For **native dbt projects**: Snowflake supports deploying dbt projects as first-class Snowflake objects via `snow dbt deploy` — the project runs on Snowflake compute, results are tracked in Snowflake, and you can schedule execution with Snowflake Tasks. dbt models can also materialize as **Semantic Views** (via the `dbt_semantic_view` package), bridging dbt transformation logic directly into Cortex Analyst. |
| 15 | Could you explain the difference between Cortex Analyst and Cortex Search? | **Cortex Analyst** answers questions about *structured* tabular data. It takes a natural language question, uses a Semantic View to understand your data model, generates SQL, and returns query results (rows and numbers). Best for: "What was revenue by region last quarter?" **Cortex Search** answers questions about *unstructured* text data (documents, PDFs, emails, support tickets). It chunks and embeds documents into a managed vector index and retrieves the most relevant passages using hybrid search. Best for: "What does our contract say about SLA penalties?" The two are often combined in a **Cortex Agent** that routes structured questions to Analyst and document questions to Search. |
| 16 | How to run Cortex using Python and Snow CLI | **Python (Snowpark ML / Cortex SDK):** `from snowflake.cortex import Complete, Summarize` — call functions directly in a Python session connected to Snowflake. Works in Notebooks, local scripts, or Streamlit apps. **Snow CLI:** `snow cortex complete "your prompt"` runs a one-off LLM completion from the terminal. `snow cortex search` queries a Cortex Search service. `snow streamlit deploy` deploys a Streamlit app. `snow sql -q "SELECT AI_COMPLETE(...)"` runs Cortex SQL functions from the command line. Install via `pip install snowflake-cli-labs` and authenticate with `snow connection add`. |
